# Команда «актуализируй проект» — регламент

Срабатывает на фразы: «p3.flow актуализируй проект», «актуализируй проект»,
«сверь состояние проекта», «что произошло по проекту». Задача — пробежаться по
всем материалам, привести `state.yaml` в соответствие с реальностью и выдать
пользователю живую сводку. Это **read-mostly** операция: правится только
`state.yaml` (и то по фактам), содержимое документов без запроса не меняется.

## Шаг 1. Прочитать заявленное состояние

```bash
python3 ~/.claude/skills/p3-flow/scripts/progress_tracker.py status
date "+%A, %d %B %Y"
```

## Шаг 2. Сверить state.yaml с фактами и поправить

Проверь каждое поле против реальных файлов:

| Поле | Как проверить | Как исправить |
|---|---|---|
| `documents.*.status` | открой документ: заполнен ли (остались ли только `_TBD_`)? not_started → а файл заполнен? | `progress_tracker.py doc <имя> <in_progress\|done>` |
| `completed_activities` | есть ли артефакты активностей (файлы циклов, разделы документов), которых нет в списке? | `progress_tracker.py complete <код>` |
| `current_activity` | согласуется ли с последними артефактами (например, есть повестка новой недели, а активность всё ещё E4)? | `progress_tracker.py set current_activity <код>` |
| `cycle_history` | папки `Docs/cycles/week-*` с заполненным закрытием, не отмеченные в истории | `progress_tracker.py cycle <id>` |

Каждое исправление — вслух: «в yaml значилось X, по факту Y — поправил».
Если факт неоднозначен (папка недели есть, но закрытие пустое) — не правь, спроси.

## Шаг 3. Что произошло в разработке (git)

Точка отсчёта — дата последней планёрки (самая свежая папка `week-*`) либо
последняя запись `cycle_history`:

```bash
git fetch --all --quiet 2>/dev/null
git branch -a --sort=-committerdate | head -10
git log --all --since="<дата последней планёрки>" --pretty="%h %an %ad %s" --date=short
git shortlog -sn --all --since="<дата>"       # кто что сделал (по числу коммитов)
git diff --stat <коммит-на-дату-планёрки>..HEAD 2>/dev/null | tail -5
```

Собери: активные ветки, кто коммитил и что по смыслу (группируй по людям и
темам, а не сырой лог), незамерженные ветки.

## Шаг 4. Новые материалы и чаты

- Новые/изменённые файлы в `Docs/` с даты планёрки:
  `find Docs -type f -newermt "<дата>" -not -path "*/.p3flow/*"`
- Появились ли новые `transcript_wN.md`, файлы в `materials/`, записи в
  `open-questions.md`, изменения в todo (отметь, какие чекбоксы закрылись).
- **Чаты:** прямого доступа к мессенджерам нет. Если экспорты/выжимки чатов
  лежат в репозитории (например, в `materials/`) — разбери новые. Иначе явно
  скажи: «к чатам доступа нет — пришлите выжимку/экспорт, включу в сводку».

## Шаг 5. Даты и планёрка

- Сегодняшняя дата и день недели.
- Дата последней планёрки = дата последней папки `week-*`.
- Ближайшая планёрка = последняя + 7 дней (если день уже прошёл — ближайший
  такой же день недели). Если в Docs зафиксирован другой ритм — используй его.

## Шаг 6. Сводка пользователю (единым блоком)

Формат:

```
📍 Стадия: <методология>, группа <X>, активность <код> — <название своими словами>
📅 Сегодня: <дата>. Последняя планёрка: <дата, N дней назад>. Ближайшая: <дата>
🔧 Поправил в state.yaml: <список исправлений или «всё сходилось»>
🌿 Разработка: <ветки, кто что сделал — по людям, 3–6 пунктов>
📄 Новые материалы: <что добавилось в Docs/ и репозиторий>
✅ Из todo недели: <сделано M из N, что именно>
❓ Открытые вопросы к планёрке: <кол-во, топ-2–3>
➡️ Следующий шаг по методологии: <что и зачем>
```

Не превращай сводку в простыню: только то, что изменилось с прошлой планёрки.
В конце предложи: продолжить со следующего шага методологии («делаем что надо
дальше») или подготовить повестку ближайшей планёрки.
